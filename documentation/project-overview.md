# PlanningAgentDemo: суть проекта

## Назначение
`PlanningAgentDemo` — демонстрационный .NET 10 проект агентной архитектуры для решения пользовательских задач через цикл:

1. построение плана (`Planner`),
2. выполнение шагов (`Executor`),
3. проверка достижения цели (`GoalVerifier`),
4. перепланирование при необходимости (`Replan`).

Проект разделен на библиотеку `PlanningAgentDemo` и тесты `PlanningAgentDemo.Tests`.

## Ключевая идея
Система не исполняет пользовательский запрос напрямую. Она сначала строит **структурированный план в JSON** (DAG шагов), затем выполняет его пошагово с учетом зависимостей и передает промежуточные данные через `Store`.

## Основные компоненты
- `Planning`: модели плана, LLM-планировщик, валидация плана.
- `Execution`: исполнитель DAG, разрешение `StoreRef`, стратегия `consume` (`auto/each/batch`), запись результатов в store.
- `Agents`: запуск LLM-шагов (`llm.chat`), парсинг строгого JSON из ответа модели.
- `Tools`: контракт инструментов (`web.search`, `web.download`) и метаданные для планировщика.
- `Verification`: проверка результата и решение `Done | Replan | AskUser`.
- `Orchestration`: полный цикл работы системы с лимитом итераций.

## Контракты данных
Все инструменты и агентные шаги используют единый envelope:

```json
{
  "ok": true,
  "data": { },
  "error": null
}
```

или

```json
{
  "ok": false,
  "data": null,
  "error": {
    "code": "string",
    "message": "string",
    "details": { }
  }
}
```

LLM-выход всегда проходит JSON extraction через `DimonSmart.AiUtils`.

## Хранилище и передача данных между шагами
- `Store` хранит промежуточные и финальные результаты в JSON.
- `StoreRef` поддерживает формы: `key`, `key[]`, `key:jsonPath`, `key:jsonPath[]`.
- Для `key[]` хранятся:
  - канонический массив `store["key"]`,
  - индексированные элементы `store["key[0]"]`, `store["key[1]"]`, ...

Это дает одновременно batch-доступ и удобную наблюдаемость.

## Инструменты в MVP
- `web.search` -> возвращает список URL.
- `web.download` -> возвращает текст страницы по URL.

Планировщик получает их описание (схемы входа/выхода, стоимость) и строит минимальный DAG под задачу пользователя.

## Что проверяют тесты
Основной unit-сценарий проверяет, что:
- план содержит логические шаги `search -> download -> compare`,
- `download` вызывается по `urls[]` в режиме `each`,
- compare-шаг получает batch-контекст `pages[]`,
- результаты корректно записываются в store, включая индексированные ключи,
- формируется структурированный `final`.

## Ограничения MVP
- до 10 шагов в плане;
- не более одного multi-input `[]` на шаг;
- поддержка базового `jsonPath` вида `$.a.b[0]`;
- фокус на корректности планирования и передачи данных, без оптимизаций производительности.
